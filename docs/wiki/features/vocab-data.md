# Vocab Data

## 목적

단어 데이터가 어떤 종류로 나뉘고, 런타임에서 어떻게 `StudyItem`으로 통합되는지 설명한다.

## 데이터 계층

### 기본 단어장

- `vocabularySets`
- `vocabularyWords`

### 주제형 단어장

- `themeWordbooks`
- `themeWords`
- 주제 토픽 단위 정렬 보유

### 비교형 단어장

- `comparisonWordbooks`
- `comparisonWords`
- `comparisonPairs`

## 런타임 집계

`src/features/vocab/model/selectors.ts`가 모든 읽기 전용 조합의 중심이다.

- 세트 조회
- 단어 조회
- 선택 가능한 단어장 목록
- `StudyItem` 변환
- 검색 텍스트 생성
- 즐겨찾기 판정

## 중요한 타입 개념

### `VocabularyWord`

일반 단어 엔트리. 기본, 주제형, 비교형 모두 이 타입을 바탕으로 움직인다.

### `StudyItem`

런타임 학습 단위.

- `word`
- `comparison`

목록, 일반 학습, 시험 일부는 `StudyItem` 기준으로 통합 처리한다.

## 세트 선택 규칙

- `all`: 모든 단어
- `favorites`: 즐겨찾기 단어
- 기본 세트 ID
- 주제형 wordbook ID
- 비교형 wordbook ID

`wrong_answers`는 정식 단어 데이터라기보다 시험 결과에서 파생되는 특수 세트다.

## 비교형 주의점

- 비교형은 한 쌍이 하나의 학습 단위
- 답안 텍스트, 검색 텍스트, 즐겨찾기 판정이 두 단어를 함께 묶는다
- 목록과 에디터에서만 직접 선택한다
- 일반 학습, 시험, 게임 후보에서는 제외한다

## 데이터 생성 경로

- 편집 원본: `src/features/vocab/editor-data/*`
- 런타임 데이터: `src/features/vocab/data/*`
- 생성 스크립트: `scripts/generate-vocab-data.mjs`

## 뜻과 중복 관리

- 같은 기본 세트 안에서 일본어 표기와 읽기가 모두 같은 항목은 하나의 엔트리로 유지한다.
- 한 단어의 여러 용법을 따로 학습해야 하더라도 표기와 읽기만으로 구분할 수 없다면 뜻 필드에 합친다.
- 비교형은 한 단어가 여러 비교 쌍에 참여할 수 있으므로 같은 단어를 별도 ID로 둘 수 있다.
- 기본 단어를 삭제하거나 통합하면 `vocabularySets[].wordIds`의 참조도 함께 제거한다.
- 뜻 수정은 편집 원본에 먼저 반영하고 생성 스크립트로 런타임 데이터를 동기화한다.

### 괄호 예시 작성

- 뜻을 구체화하는 대상이나 상황은 `(바닥을) 닦다`, `(비에) 젖다`처럼 필요한 조사까지 괄호 안에 넣는다.
- 서로 다른 조사가 필요한 예시는 `(입으로) 불다, (악기를) 불다`처럼 따로 적는다.
- 같은 조사로 나열하는 대상은 `/` 대신 `·`를 사용한다.
- `(의태어) 두근거리다`처럼 분류 설명 뒤에는 한 칸을 띄운다.
- 번역투 설명보다 실제 한국어 결합 표현을 우선한다.

## 수정 시 주의점

- 에디터 구조를 바꾸면 런타임 selectors와 생성 스크립트도 함께 봐야 함
- 검색, 학습, 시험은 대부분 `StudyItem`을 읽으므로 단순 JSON 수정으로 끝나지 않을 수 있음

## 표기와 읽기 검증

- 전체 단어장을 점검할 때는 일본어 표기와 읽기의 어미 일치, 사전 표제어·읽기 조합, 품사와 동사 자·타 구분을 함께 확인한다.
- 단독 표제어의 읽기는 연탁된 접미 형태가 아니라 독립된 읽기를 사용한다. 예: `箱`은 `はこ`.
- 사전마다 자·타 구분이 엇갈리는 동사는 기계적으로 바꾸지 않고, 분류가 명확한 항목만 수정한다.
- 편집 원본을 수정한 뒤 생성 스크립트를 실행하고, 데이터 품질 테스트로 확정된 수정과 어미 일치를 회귀 검증한다.
